# UI Content Audit 2026-03-07

## Мета

Після виправлення page navigation і natural sorting виконано окрему перевірку на реальному Cisco-контенті з каталогу `static/books`.

---

## Що перевірялося

- порядок сторінок у дереві книг для англомовних і російськомовних курсів
- робота natural sorting для нумерації виду `0.0.1.1 ... 0.0.1.11`
- наявність аномалій у файлових іменах, що можуть впливати на UI
- можливі колізії після нормалізації шляхів
- узгодженість page index і navigation logic

---

## Підсумок

Поточні виправлення працюють коректно.

Зокрема:

- порядок `0.0.1.1 ... 0.0.1.11` у `Introduction to Networks v6 (ITN) Extended Ru/Глава 0. Знакомство с курсом` відновлено
- `prev/next` тепер можуть будуватися через server-side navigation context
- колізій шляхів після нормалізації не виявлено

---

## Виявлені аномалії контенту

У вихідних даних книг знайдено:

- `16` шляхів з початковими пробілами в імені
- `7` шляхів із кінцевими пробілами в імені
- `0` колізій після нормалізації `strip()` по сегментах шляху

Типові приклади:

- `Introduction to Networks v6 (ITN) Extended Ru/Глава 0. Знакомство с курсом/ 0.0.1.2 Мировое сообщество `
- `Introduction to Networks v6 (ITN) Extended Ru/Глава 0. Знакомство с курсом/ 0.0.1.7 Инженерно-т___ические журналы `
- `Routing and Switching Essential v6 (RSE) Extended Ru/Глава 1 Концепция маршрутизации/ 1.1.4.6 Лабораторна___(CLI) системы Cisco IOS`

Такі імена не є помилкою коду застосунку, але є "брудними" даними і можуть впливати на:

- візуальну акуратність дерева
- переносимість контенту між платформами
- ручне адміністрування контенту

---

## Що вже компенсується кодом

Поточний код уже компенсує ці проблеми на рівні UI/навігації:

- natural sorting ігнорує крайні пробіли при побудові sort key
- page index сортується по сегментах шляху
- navigation lookup нормалізує шляхи через обрізання крайніх пробілів

Тобто UI тепер не ламається від цих аномалій.

---

## Залишковий ризик

Ризик лишається не в логіці застосунку, а в самих файлах контенту.

Якщо в майбутньому:

- контент буде переноситися між файловими системами
- книги будуть імпортуватися або перейменовуватися вручну
- з'явиться web-адмінка або масовий імпорт

то варто додати окремий етап нормалізації імен файлів і папок.

---

## Рекомендація

Наступний розумний крок для роботи з контентом:

- додати окремий dry-run інструмент аудиту/нормалізації імен директорій та файлів у `static/books`

Це не є критичним для поточної працездатності, але корисне для довгострокової підтримки бібліотеки.
